## Code change (non-engagement fallback) — playlist-id validation
**Root cause:** 17 topics carry a **structurally-invalid 13-char placeholder** `playlist_yt_id`
(a real user-created YouTube playlist id is `PL`+32 = 34 chars). Every `playlistItems.insert` for
those topics 404s ("Playlist not found"), wasting 50 quota/attempt and leaving videos out of their
playlist. The existing post-publish self-heal (`is_playlist_missing` → null → recreate) is correct
for *deleted* playlists (I verified the real `add_to_playlist → _classify → is_playlist_missing`
chain returns `True` on a 404), but these placeholders predate it and never got recreated —
today's publishes of vid 565 (t24) and 568 (t25) both re-hit the 404 with `added_to_playlist=0`.

**Fix** (2 files + test): `youtube.is_valid_playlist_id()` (structural check) and, in
`publish_loop._publish_one` **before upload**, drop a structurally-invalid stored id and route through
`ensure_topic_playlist` — the same path that produced the working 34-char playlists (and which
self-commits). No `Video.status` transition touched; degrades safely (if recreate fails, `pl` stays
None → description/comment omit the link, `playlist_add` is skipped, publish still succeeds).

**Verified (synchronous):** `tests/verify_publish.py` **30/30** incl. 5 new checks (invalid id → topic
remapped to a freshly-created valid playlist, invalid mapping dropped, `add_to_playlist` called with
the valid id, video still publishes); `tests/verify_storyboard.py` **40/40**; imports OK; manager
restarted, `/api/dashboard` = **200**.
**Watch next run (live-only effect):** both channels were done publishing today and near quota cap,
so the real recreate for the 17 topics fires on each topic's *next* publish — confirm via DB that a
34-char `playlist_yt_id` replaces the 13-char placeholder as those topics publish, and that
`playlist_add` errors go to 0.
