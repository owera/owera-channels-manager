## Triage
**One root cause, and it corrects yesterday's narrative.** ch2's 8 failed uploads (448–455,
all `has_file:true`, all "upload repeatedly stalled … gave up re-queuing") were **not** a network
stall, not file corruption, and **not** the timeout code (that was a separate self-inflicted ch1
regression, already reverted in `acb30ea`). The real cause:

> **ch2's OAuth refresh token is revoked.** Confirmed **synchronously** by calling
> `youtube.get_service('rodrigo-recio')` directly → `RefreshError: invalid_grant: Token has been
> expired or revoked`, while `owera-software` returns a working service (live API call OK). The
> token died between ch2's last success (07-01 09:05 UTC) and the first publish failure after it.

**Why it was invisible (the bug I fixed — see Engagement/Code below):** `_load_creds` let the
`RefreshError` propagate out of `get_service`. The publish loop only catches `NeedsConnect`, so the
error escaped `_publish_one` with the video already in `PUBLISHING`; the stuck-publishing recovery
loop then **mislabeled it a "stall"**, recycled it to the retry cap, and marked it `FAILED` — while
`channel.oauth_status` stayed `"connected"`. So the dashboard showed ch2 healthy, the issues digest
`oauth:[]` was empty, no ⚠ ever fired, and the drip kept burning on doomed uploads. Admin endpoints
also 500'd on the same leak.

**Auto-fixes this run:** none were possible — every terminal item needs the operator:
- ch2 oauth revocation → **escalated** (operator reconnect). I proactively flipped the DB state
  live so the app now shows it: `GET /api/channels/2/oauth-status` → `oauth_status:"expired"`, and
  `GET /api/agent/issues` now returns the `oauth` escalation (`needs_operator` 9→10). ch1 re-check
  stayed `connected`. **Observed after-state, quoted, not assumed.**
- 8 failed ch2 videos (448–455) → **left `FAILED`**, not retried. Retrying is futile while the token
  is dead (they'd only wait as APPROVED, or under the old bug re-stall). They're <7d old and are
  good content — do **not** delete; retry them after reconnect.
- ch1 quota 8500/9000 → informational (`auto:false`, "monitor"). ch1 published its 5-video daily
  budget; near-cap is expected and healthy. No action.

### Code change (the one change this run) — shipped `362691a`
`app/services/youtube.py` `_load_creds`: catch `google.auth.exceptions.RefreshError` on
`creds.refresh()` and return `None`, so `get_service` raises the clean `NeedsConnect` every caller
already handles.
- **Lifecycle trace (state machine touched — required):** `_load_creds` ← `get_service` ← {publish
  loop, metrics loop, analytics loop, topic_playlist, youtube_admin, thumbnail/playlist}.
  *Before:* revoked token → `RefreshError` leaks → in `_publish_one` the `except NeedsConnect`
  misses it → video stuck `PUBLISHING` → recovery loop recycles → `FAILED` (mislabeled "stall"),
  `oauth_status` never updated; admin endpoints 500.
  *After:* `get_service` raises `NeedsConnect` → `_publish_one` sets `oauth_status=EXPIRED` and
  returns the video to `APPROVED`; the publish tick's `if oauth_status != CONNECTED: continue`
  then **skips** the channel (no more doomed uploads); the issues digest `oauth != connected`
  branch fires (operator escalation); admin endpoints return a clean 409. metrics/analytics/
  topic_playlist already `except Exception`, so they're unaffected. Valid tokens (ch1) return on
  `creds.valid` before `refresh()` is ever called — path untouched.
- **Verified (guardrail 3, synchronous):** import OK; `get_service('owera-software')` + live
  `channels().list` OK (no regression); `get_service('rodrigo-recio')` now raises `NeedsConnect`;
  simulated `_publish_one` handler flips the channel to `oauth_status=expired`. Post-restart, live:
  dashboard 200, ch2 re-check → `expired`, ch1 → `connected`, digest now carries the oauth
  escalation.
