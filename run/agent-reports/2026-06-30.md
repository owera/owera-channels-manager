## 4. Improve the app — 1 code change (self-heal stale playlists)

**Why:** the `playlist_add` 404 is a real recurring pipeline bug (logs, not just topic 7):
if *any* topic's playlist is deleted on YouTube, the publish loop spams 404s on every
future publish for that topic with no recovery, because the recreate safety-net only
triggers when `playlist_id` is unset — not when it's set-but-dead.

**Lifecycle trace (per guardrail — this does NOT change any `Video.status` transition):**
the change lives entirely in the post-upload, best-effort playlist step. The video is
already `PUBLISHED` before this code runs (`status` set at publish_loop.py:148); the
playlist add is wrapped in try/except and never alters video state. The only state touched
is `Topic.playlist_id` (set to `None`), which feeds the *existing* `if not
topic.playlist_id: ensure_topic_playlist(...)` recreate path on the next publish. No loop
routing changes.

**Change:**
- `app/services/youtube.py`: new `is_playlist_missing(e)` — True only for a genuine
  playlist-gone error (reason `playlistNotFound` or bare HTTP 404), False for everything
  else (so a transient/quota error never destroys a valid mapping).
- `app/services/publish_loop.py`: on a playlist-add failure, if `is_playlist_missing(e)`,
  drop the stale `topic.playlist_id` so the next publish auto-recreates the playlist.

**Verification (guardrail 3 — behavior, not just boot):**
- `uv run python -c "import app.main"` → imports clean.
- **Unit-exercised the new gate** (the logic that decides whether self-heal runs):
  - `playlistNotFound` 404 → `True` ✓ (self-heal fires)
  - `quotaExceeded` 403 → `False` ✓ (mapping preserved — no false self-heal)
  - transient 500 → `False` ✓
  - non-`HttpError` exception → `False` ✓
  - all assertions passed.
- Service restarted (`launchctl kickstart -k`), `GET /api/dashboard` → 200 on first try.
- **Honest limit:** the full end-to-end recreate-on-next-publish path can't be force-fired
  today — ch1's publish budget is already spent (5/5) and forcing an upload would burn real
  quota and publish a real video. The detection gate is unit-verified and the recovery
  reuses the already-battle-tested `ensure_topic_playlist`. The end-to-end effect will be
  observable on topic 7's next publish — flagged in §6 to confirm next run.
